# 🐍 Задания для интервью QA Automation (Python)

В данном репозитории собрана база практических и теоретических заданий для проведения технических интервью на позицию **QA Automation Engineer (Python)** уровней Middle / Senior.

Идея создания этой обновляемой базы знаний родилась из потребности в глубокой и качественной проверке хард-скиллов кандидатов. 
Здесь **нет** заученных вопросов из классических чек-листов (вроде "назовите принципы ООП" или "что такое фикстура"), которые проверяют лишь умение гуглить перед собеседованием.

Напротив, большинство заданий:
* Построены на реальном коде (ревью, поиск ошибок, предсказание поведения).
* Заставляют кандидата рассуждать и демонстрировать глубину понимания внутренних механизмов фреймворков.
* Не всегда имеют единственный правильный ответ, позволяя оценить архитектурный опыт специалиста.

## 🛠 Технологический стек

Вопросы и задания охватывают современный стек Python-автоматизации:

**✅ Уже реализовано:**
* **Pytest** (фикстуры, маркеры, хуки, параметризация, плагины)
* **Pydantic** (валидация данных, схемы)
* **Playwright** (UI-автоматизация, паттерны Page Object, ожидания)

**🕒 В планах (будет добавлено позже):**
* **Requests / HTTP** (REST API, сессии, моки)
* **GitLab CI** (пайплайны, джобы, артефакты)
* **Docker** (контейнеризация тестов, docker-compose)
* **Allure Framework** (репортинг, интеграция)

## 📂 Структура репозитория

Каждый логический блок или фреймворк вынесен в отдельную директорию в корне проекта (например, `pytest-base/`).

Внутри блоков структура делится по темам. Стандартная структура одного задания выглядит так:

```text
📁 pytest-base/
 └── 📁 markers/
      └── 📁 builtin/
           ├── test_builtin_markers.py  # Файл с кодом для кандидата
           └── guide.md                 # Шпаргалка для интервьюера
```

## 👨‍🏫 Регламент использования (Для интервьюера)

База заданий спроектирована так, чтобы её было легко применять прямо на техническом скрининге (например, через Share Screen в Яндекс.Телемост, Zoom или любой другой платформе для онлайн-интервью).

1. **Выбор заданий:** Перед интервью выберите 3-4 задания из разных блоков, соответствующих грейду кандидата.
2. **Демонстрация:** Покажите кандидату **только** `.py` файл (или фрагмент кода из него). Ни в коем случае не показывайте `guide.md`.
3. **Постановка задачи:** Попросите кандидата вслух проанализировать код. Задайте вопросы, описанные в блоке `> Вопросы:` в вашем `guide.md`.
4. **Оценка:** Сверяйте рассуждения кандидата с блоком `Правильные ответы и объяснения` в вашем `guide.md`. Обращайте внимание не только на правильный ответ, но и на то, как кандидат к нему пришёл.

*Примечание: Задания не предполагают обязательного live-coding. Основной упор делается на код-ревью и архитектурный анализ.*

## ✍️ Инструкция по созданию новых заданий

Приветствую пополнение базы новыми интересными кейсами из вашей практики! Чтобы репозиторий оставался консистентным, соблюдайте следующий формат:

### 1. Создание файлов
Для новой темы создайте новую папку внутри соответствующего раздела. Папка должна содержать минимум два файла:
* `test_<тема>.py` (или другой уместный формат) — код для кандидата.
* `guide.md` — инструкция и ответы для проверяющего.

### 2. Оформление кода (`.py`)
* Код должен быть рабочим (или содержать умышленную ошибку, которую нужно найти).
* Код должен быть лаконичным (убирайте лишние импорты и бизнес-логику, не относящуюся к сути вопроса).
* Никаких подсказок, комментариев с ответами или спойлеров в самом коде! Код должен выглядеть как обычный кусок продакшена.

### 3. Оформление `guide.md` (Шаблон)

```markdown
# Название темы

> Краткое описание того, что нужно дать/показать кандидату.
> 1. Вопрос кандидату номер 1?
> 2. Вопрос кандидату номер 2?

### Правильные ответы и объяснения:

1. **Ответ на первый вопрос:** 
   Развернутое объяснение того, что должен сказать кандидат. Почему код работает именно так, ссылки на внутренние механизмы фреймворка.

2. **Ответ на второй вопрос:**
   Развернутое объяснение...
```

### 4. Чего стоит избегать при составлении
* **Тривиальных вопросов.** Не нужно спрашивать "Зачем нужен assert?".
* **Громоздкого кода.** Кандидат должен суметь прочитать и осознать код с экрана за 1-2 минуты.
* **"Синтаксического" снобизма.** Вопрос должен проверять логику и архитектуру, а не знание того, в какой версии Python появился конкретный метод у строки.
