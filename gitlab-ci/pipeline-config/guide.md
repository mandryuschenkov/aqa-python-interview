# Конфигурация GitLab CI/CD для пайплайна автотестов

> 🟡 **Вопросы (Middle):**
> Перед вами реальный файл `.gitlab-ci.yml` из проекта автоматизации UI-тестов на Playwright. Изучите его и ответьте на вопросы:
>
> 1. Переменные `ENV_NAME` и `COMMAND` заданы в формате с `value` и `description`, а `TZ` — просто `KEY: VALUE`. В чём разница?
> 2. Разберите, что происходит в блоках `before_script`, `script` и `after_script`.
>    - В каком порядке выполняются эти блоки?
>    - Будут ли запущены два блока при падении первого?
>    - Почему `source .env` встречается трижды?
> 3. Какие три источника запуска покрываются в блоке `rules`? Что произойдёт, если пайплайн запустить по `merge request`?
> 4. Что будет выгружено в качестве артефактов и по каким правилам? Что случится, если после прогона какого-то из заявленных артефактов (например, `screenshots`) не окажется в директории?

> 🔴 **Вопросы (Senior):**
> 1. Что делает `eval ENV_FILE=\$"${ENV_NAME}"` и как работает `grep -v '^#' "$ENV_FILE" > .env`? Зачем нужен `eval` вместо прямого подставления `${ENV_NAME}`?
> 2. Что делает `export TMS_TEST_RUN_NAME="Autotests WEB UI $(date -d "$CI_JOB_STARTED_AT" '+%d-%m-%Y %H:%M:%S')"`? Как работает shell-подстановка `$(...)` и как `export` влияет на выполнение `$COMMAND`?

### Ответ:

**Middle:**
1. Формат с `value` и `description` создаёт **UI-переменную** — она отображается в форме запуска пайплайна в GitLab, позволяя менять значение из веб-интерфейса. Простой `KEY: VALUE` — статичная переменная, видимая только внутри пайплайна.
2. **Порядок выполнения:** `before_script` → `script` → `after_script`.
   - При падении `before_script` блок `script` **не запустится**, но `after_script` **запустится**.
   - `source .env` нужен в каждом блоке, потому что каждый блок выполняется в **новой shell-сессии** — переменные из предыдущего блока не сохраняются.
3. Покрываются: `web` (ручной запуск из UI), `schedule` (по расписанию), `trigger` (из другого пайплайна). По `merge request` пайплайн **не запустится** — нет правила для `merge_request_event`.
4. Будут выгружены директории `logs`, `test-results`, `screenshots` с TTL 1 неделя. `when: always` означает сбор артефактов **всегда** — при любом статусе джобы. Если артефакта нет — GitLab выведет **warning**, но пайплайн **не упадёт**.

### Пояснение:

**Middle:**
1. **UI-переменные vs статичные переменные**:
   * Формат `ENV_NAME: { value: DEV, description: "Выберите окружение" }` — это **interactive variable**. В GitLab UI при запуске пайплайна появляется форма, где кандидат может выбрать значение или ввести своё.
   * `TZ: Europe/Moscow` — обычная переменная, доступна только внутри пайплайна, без UI-формы.
   * `COMMAND` с `value: pytest --testit --reruns 2 tests` позволяет изменить команду pytest без правки YAML — например, добавить `-n 4` для параллельного запуска.

2. **Жизненный цикл джобы (`before_script` → `script` → `after_script`):**
   * Каждый блок выполняется в **новой оболочке** (shell-сессии). Это ключевой момент: переменные, установленные в `before_script` (включая `source .env`), **не доступны** в `script` и `after_script`. Поэтому `source .env` повторяется трижды.
   * **Порядок выполнения строго линейный:** `before_script` → `script` → `after_script`.
   * **При падении `before_script`:** `script` **не выполняется**, но `after_script` **выполняется**. Это намеренный дизайн GitLab CI — `after_script` создан для очистки и отчётности, даже если подготовка провалилась.
   * **При падении `script`:** `after_script` **выполняется**. Это основное сценарное использование — например, отправка отчёта в Telegram после прогона тестов.
   * `python env_setup.py` в `before_script` — скрипт инициализации окружения. Он может делать то, что нельзя выразить в `.env`: создавать директории, настраивать зависимости, инициализировать базы данных.
   * `python tg_report.py` в `after_script` — отправка отчёта о результатах тестов (например, в Telegram). Вынесена в `after_script`, чтобы срабатывать независимо от результата прогона.

3. **`rules` — источники запуска:**
   * `web` — ручной запуск из веб-интерфейса GitLab (кнопка "New pipeline").
   * `schedule` — пайплайн по расписанию (cron), например, ночной прогон.
   * `trigger` — запуск из другого пайплайна (вложенные/триггерные пайплайны). Используется для оркестрации: один пайплайн запускает другой.
   * `- when: manual` — во всех остальных случаях джоба создаётся в пайплайне, но требует ручного подтверждения кнопкой "Run".
   * **Merge request не покрыт** — нет правила `if: $CI_PIPELINE_SOURCE == "merge_request_event"`. По MR пайплайн не запустится.

4. **Артефакты:**
   * `paths: [logs, test-results, screenshots]` — директории, которые GitLab сохранит после завершения джобы.
   * `when: always` — артефакты собираются **всегда**, при любом статусе джобы (passed, failed, skipped, canceled). Без этого параметра GitLab по умолчанию собирает артефакты только при успешном завершении.
   * `expire_in: 1 week` — артефакты удаляются через неделю, освобождая место на CI-сервере. Без него артефакты хранятся **бессрочно**, что при регулярных прогонах быстро заполнит хранилище.
   * **Если артефакта нет:** GitLab выводит warning в логах, но **не прерывает пайплайн**.

**Senior:**
1. `eval` и `grep`:
   * `eval ENV_FILE=\$"${ENV_NAME}"` — без `eval` присваивание `ENV_FILE=\$"${ENV_NAME}"` запишет в переменную строку `"DEV"`, а не значение. `eval` выполняет строку как shell-команду: сначала `${ENV_NAME}` раскрывается в `DEV`, затем `eval` присваивает `ENV_FILE=DEV`.
   * `grep -v '^#' "$ENV_FILE" > .env` — `-v` инвертирует поиск, `'^#'` — строки, начинающиеся с `#`. Фильтрует комментарии в файле окружения.

2. `export` и `$(...)`:
   * `export TMS_TEST_RUN_NAME="..."` — переменная становится доступной для дочерних процессов (включая `$COMMAND` — pytest). Без `export` переменная видна только в текущей shell-сессии.
   * `$(date -d "$CI_JOB_STARTED_AT" '+%d-%m-%Y %H:%M:%S')` — `$(...)` — shell-подстановка: сначала выполняется команда `date`, которая парсит ISO-8601 timestamp из встроенной переменной GitLab `CI_JOB_STARTED_AT` и форматирует в читаемый вид, затем результат подставляется в строку.
   * `$COMMAND` выполняется как shell-команда. `export` делает `TMS_TEST_RUN_NAME` доступной для pytest-плагина testit, который использует её для именования тестовых прогонов в TMS.

### Навыки:
* **Понимание жизненного цикла джобы:** порядок выполнения блоков, поведение при падениях, изоляция shell-сессий.
* **UI-переменные GitLab:** разница между интерактивными и статичными переменными, управление окружением из веб-интерфейса.
* **Rules и источники запуска:** знание триггеров (`web`, `schedule`, `trigger`), понимание gaps в покрытии (merge request).
* **Артефакты:** `when: always`, `expire_in`, поведение при отсутствии файлов, awareness про `fail_on_missing_artifacts`.
* **Shell-сессии в CI:** понимание, что каждый блок — отдельная оболочка, и почему `source .env` повторяется.
* **Shell-скриптинг (Senior):** `eval`, `grep`, `export`, `$(...)`, понимание разницы между присваиванием и выполнением команды.
