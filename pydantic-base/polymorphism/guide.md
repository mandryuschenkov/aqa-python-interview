# Полиморфизм и объединение классов для типизации в Pydantic

> 🔴 **Вопросы (Senior):**
> Представлен код с использованием объединения типов в Pydantic. 
> В словаре `api_response` передается `type: "image"`, но отсутствует поле `url`.
> 
> 1. Возникнет ли ошибка валидации из-за отсутствия поля `url` при вызове `model_validate`?
> 2. Объектом какого класса будет инициализировано свойство `chat.message` и какое значение примет его атрибут `type`?
> 3. Почему такая реализация не обеспечивает строгую маршрутизацию парсинга по значению поля `type`?
> 4. Как технически правильно реализовать эту схему для строгой валидации в зависимости от значения поля `type`?

### Ответ:
`ValidationError` не возникнет, `assert` выполнится успешно. `chat.message` станет объектом `TextMessage` со значением атрибута `type="image"`. Значение по умолчанию не является ограничением при валидации (Pydantic проверяет лишь соответствие типу `str`). Для строгой маршрутизации необходимо использовать `Literal` типы и `discriminator`.

### Пояснение:
1. **Отсутствие ValidationError:** Pydantic валидирует `Union` слева направо. Первой проверяется модель `TextMessage`. Входящий словарь содержит строковое поле `type` и поле `text`. Это полностью удовлетворяет аннотациям `TextMessage`. Модель `ImageMessage` (и отсутствие в ней `url`) даже не рассматривается алгоритмом валидации.
2. **Тип объекта:** Результатом парсинга будет инстанс `TextMessage(type='image', text='Custom text message')`. Входящее значение `"image"` просто перезаписывает дефолтное значение `"text"`, так как оба являются типом `str`.
3. **Механика Union:** Дефолтное значение поля (например, `= "text"`) используется только при отсутствии ключа во входящих данных. Оно не накладывает ограничений на то, какое значение может быть передано. Pydantic проверяет совпадение по типу (`str`), поэтому пропускает любую строку.
4. **Правильная реализация (Tagged Union):**
   Необходимо жестко зафиксировать допустимые значения через `Literal` и явно указать Pydantic ключ для дискриминации.
   ```python
   from typing import Literal
   from pydantic import BaseModel, Field

   
   class TextMessage(BaseModel):
       type: Literal["text"]
       text: str

   
   class ImageMessage(BaseModel):
       type: Literal["image"]
       url: str
       text: str | None = None

   
   class ChatPayload(BaseModel):
       message: TextMessage | ImageMessage = Field(discriminator="type")
   ```
   В этом случае при передаче `"type": "image"` Pydantic сразу направит парсинг в модель `ImageMessage` и выбросит корректный `ValidationError: Field required [url=missing]`.

### Навыки:
* Понимание алгоритмов разрешения типов при использовании `Union` в Pydantic.
* Отличие поведения аннотаций типов (`str`) от ограничений по значению (`Literal`).
* Навык реализации полиморфных структур данных (Tagged Unions / Discriminators).
