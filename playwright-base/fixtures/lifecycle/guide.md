# Изоляция тестов и переиспользование состояния

> Перед вами код с двумя тестами, использующими фикстуру `page`. В первом эмулируется логин (токен в `localStorage`). Во втором проверяется наличие токена.
> 
> 1. По какой причине упадёт второй тест?
> 2. Каким образом мы можем доработать тесты или конфигурацию фреймворка, чтобы второй тест прошел успешно и смог переиспользовать авторизацию из первого?
> 
> Дополнительный вопрос, если кандидат предложил несколько вариантов: Сравните подход с сохранением `storage_state` и подход с использованием единого глобального `BrowserContext`. В каких случаях каждый из них предпочтительнее?

### Правильный ответ:
Второй тест завершится ошибкой `AssertionError` (токен равен `None`), так как по умолчанию тесты полностью изолированы. 

### Объяснение (что должен рассказать кандидат):

1. **Проблема второго теста (Изоляция):**
   Для каждой тестовой функции `pytest-playwright` автоматически создает новый, полностью очищенный `BrowserContext`. Состояние (`cookies`, `localStorage`) уничтожается при закрытии контекста первого теста и не переносится в следующие тесты.

2. **Решение (Переиспользование состояния):**
   Кандидат должен предложить механизм `storage_state`. Авторизация выполняется один раз. Состояние браузера сохраняется в JSON-файл (`context.storage_state()`) и применяется к последующим тестам путем переопределения встроенной фикстуры `browser_context_args`.

3. **Сравнение подходов (к дополнительному вопросу):**

   * **Подход А: `storage_state` (Рекомендуемый подход)**
     * **Преимущества:** Полная изоляция тестов (каждый тест получает независимую страницу с исходным состоянием DOM). Поддержка безопасного параллельного выполнения (например, через `pytest-xdist`).
     * **Недостатки:** Требуется дополнительная логика для обработки устаревания токенов во время длительных прогонов. Не сохраняет состояние активных сетевых соединений (например, WebSocket).
   
   * **Подход Б: Persistent Context / Session Scope**
     Создается единый глобальный контекст браузера, в котором последовательно выполняются все тесты.
     * **Преимущества:** Меньшие накладные расходы времени на инициализацию перед каждым тестом. Удобство при отладке сложных непрерывных сценариев, состоящих из цепочки тестов.
     * **Недостатки:** Нарушение изоляции (побочные эффекты от первого теста могут привести к падению последующих). Невозможность безопасного распараллеливания тестов (вкладки будут конфликтовать за фокус и элементы DOM).

### Что показывает эта задача:
Понимание базовой концепции `BrowserContext` во фреймворке Playwright и умение кандидата решать архитектурные задачи, связанные с переиспользованием состояния авторизации между изолированными тестами.
