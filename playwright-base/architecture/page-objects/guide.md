# Паттерн Page Object в Playwright

> 🟡 **Вопросы (Middle):**
> Перед вами код тестового класса `LoginPage`, реализующего популярный паттерн Page Object Model. 
> 
> 1. Проведите код-ревью этого класса. Какие антипаттерны, принесенные из других фреймворков (например, Selenium), вы здесь видите?
> 2. Как бы вы переписали этот класс, чтобы он соответствовал официальным рекомендациям и архитектуре Playwright?

### Ответ:
Класс написан в стиле Selenium: локаторы хранятся как строки, а объекты `Locator` создаются заново при каждом вызове метода. В Playwright рекомендуется создавать объекты `Locator` один раз в конструкторе класса (`__init__`) и переиспользовать их.

### Пояснение:

1. **Проблемы текущей реализации (Антипаттерны):**
   * **Хранение локаторов в виде строк:** В классе объявлены строковые константы (`USERNAME_INPUT = "#username"`). Это лишает разработчика преимуществ строгой типизации и автодополнения в IDE.
   * **Создание локаторов "на лету":** В методах действий (`enter_username`, `click_submit`) каждый раз вызывается `self.page.locator(...)`. Это избыточно и не позволяет использовать мощный механизм цепочек локаторов, когда один локатор ищется внутри другого.
   * **Излишняя гранулярность методов:** Создание отдельных методов для каждого атомарного действия (`enter_username`, `enter_password`, `click_submit`) раздувает код. В Playwright принято объединять логически связанные действия в один бизнес-метод (например, `login(username, password)`).

2. **Рекомендуемый подход (Playwright-way):**
   Кандидат должен предложить перенести инициализацию локаторов в конструктор `__init__`, сохраняя их как свойства экземпляра класса. Идеальный вариант рефакторинга выглядит так:

   ```python
   class LoginPage:
       def __init__(self, page: Page):
           self._page = page
           # Определяем корневой элемент формы по aria-label
           self._root = page.get_by_label("Login form")
           
           # Остальные локаторы ищутся строго внутри корневого
           self.username_input = self._root.get_by_label("Username")
           self.password_input = self._root.get_by_label("Password")
           self.submit_button = self._root.get_by_role("button", name="Log in")

       # Атомарные действия объединяются в бизнес-шаги
       def login(self, username: str, password: str):
           self.username_input.fill(username)
           self.password_input.fill(password)
           self.submit_button.click()
   ```

3. **Преимущества такого подхода:**
   * **Инкапсуляция:** Определение единого `self._root` сужает область поиска для всех остальных элементов. Это защищает от ложных срабатываний, если на странице (например, в хедере) появится еще одно поле "Username".
   * **Переиспользование:** Объекты `Locator` создаются один раз при инициализации.
   * **Читаемость:** Код тестов становится короче и выразительнее, скрывая низкоуровневые взаимодействия за понятными бизнес-методами.

### Навыки:
Умение кандидата адаптировать классические паттерны проектирования под архитектурные особенности конкретного инструмента. Задача отлично выявляет инженеров, которые просто переносят свои привычки из Selenium в Playwright, не изучая официальную документацию и лучшие практики нового фреймворка.
