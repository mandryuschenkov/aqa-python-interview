# Слияние параметров сессии (requests.Session)

> [!CAUTION]
> **Вопросы (Senior):**
> Перед вами код, где при инициализации клиента на уровне `requests.Session` заданы глобальные `params`, `headers` и `auth`. В методе `get()` передаются локальные параметры, а URL уже содержит query-строку.
> 
> 1. Как будет выглядеть итоговая строка параметров в URL?
> 2. Какие заголовки будут в финальном запросе?
> 3. Какие учетные данные (`auth`) будут использованы?

### Ответ:
1. **URL:** `.../anything?filter=active&api_version=v2&limit=100`. Локальные параметры имеют приоритет.
2. **Заголовки:** `X-Source` унаследуется из сессии, заголовок `User-Agent` будет полностью удален.
3. **Auth:** Запрос уйдет с локальными данными `guest` (кортеж полностью заменяется).

### Пояснение:

1. **Слияние Query-параметров:**
   Сборка параметров происходит из трех источников (от низшего приоритета к высшему):
   * `session.params` (`api_version=v1`).
   * Строка в `url` (`filter=active`). 
   * Локальный аргумент `params` (`api_version=v2`, `limit=100`).
   Локальный словарь переопределяет сессионный. Результирующая строка `api_version=v2&limit=100` приклеивается к уже существующей в URL части `?filter=active`.

2. **Объединение заголовков и значение `None`:**
   Словари `session.headers` и локальные `headers` объединяются. Ключ `X-Source` переходит в запрос. Передача значения `None` для ключа `User-Agent` дает команду удалить этот заголовок из запроса.

3. **Слияние Auth:**
   Кортежи `auth` не объединяются. Локальный аргумент `auth` полностью заменяет сессионный `session.auth`.

### Навыки:
Знание внутренних механизмов слияния параметров библиотеки `requests`. Умение предсказывать поведение клиента при конфликте конфигураций на уровне сессии и конкретного запроса. Проверка понимания разницы в обработке словарей (headers, params), кортежей (auth) и специальных значений (`None` для удаления заголовка).
