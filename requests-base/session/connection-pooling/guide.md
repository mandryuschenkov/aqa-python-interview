# Пул соединений (requests.Session vs requests.get)

> [!WARNING]
> **Вопросы (Middle):**
> Перед вами две функции, которые последовательно скачивают данные 1000 пользователей по API. Первая использует `requests.get()`, вторая — `requests.Session()`.
> 
> 1. Какая из функций отработает быстрее и почему?
> 2. Какой механизм сетевого протокола используется во второй функции для оптимизации?

> [!CAUTION]
> **Дополнительный вопрос (Senior):**
> Что произойдет, если сервер перестанет отвечать?

### Ответ:
Функция `fetch_users_with_session` отработает быстрее. `requests.Session` поддерживает пул соединений и HTTP Keep-Alive, что позволяет переиспользовать TCP/TLS-соединения для запросов к одному хосту.

### Пояснение:

1. **Разница в производительности:**
   * **`requests.get()`:** Открывает и закрывает TCP-соединение на каждом запросе, выполняя затратные процедуры handshake и TLS-согласования.
   * **`requests.Session()`:** Устанавливает соединение при первом запросе и сохраняет его в пуле (через `urllib3`). Последующие запросы используют уже открытый канал.

2. **Механизм под капотом (HTTP Keep-Alive):**
   Используется механизм HTTP/1.1, при котором клиент и сервер не закрывают сокет после передачи ответа, оставляя его открытым для будущих запросов.

3. **Проблема таймаутов:**
   В библиотеке `requests` по умолчанию отсутствует таймаут на ожидание ответа (`timeout=None`). Если сервер перестанет отправлять данные, выполнение функции заблокируется навсегда, что приведет к зависанию прогона.

### Навыки:
Понимание разницы между атомарными запросами и сессиями. Знание основ сетевого взаимодействия (TCP/IP, TLS, Keep-Alive) и специфики работы с дефолтными настройками таймаутов в библиотеке `requests`.
