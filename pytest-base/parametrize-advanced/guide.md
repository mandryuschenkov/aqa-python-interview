# Продвинутая параметризация фикстур

> Перед вами код с фикстурой уровня `module` и тестовым классом. Каждая тестовая функция параметризована через `indirect=True` по фикстуре `db_connection`.
> 
> * Сколько всего раз будет вызван код до `yield` в фикстуре `db_connection` (вывод в лог "Open connection...")?
> * В каком порядке отработают тесты и фикстуры?
> * Объясните, почему `scope="module"` ведет себя именно так в контексте параметризации.

### Правильный ответ:
Код до `yield` (открытие подключения) будет вызван **2 раза** (один раз для параметра `"admin"` и один раз для `"guest"`). Всего сгенерируется 5 тестов (по два на первые две тестовые функции, и один на последнюю).

Pytest отсортирует выполнение тестов так, чтобы минимизировать пересоздание фикстуры, сгруппировав их по значению параметра.

### Объяснение (что должен рассказать кандидат):

1. **Суть `indirect=True`:**
   Параметры `["admin", "guest"]` передаются не напрямую как аргументы в тестовые функции, а отправляются внутрь фикстуры `db_connection` через `request.param`. Фикстура выполняет логику (в данном случае "подключается к БД") и возвращает готовый объект.

2. **Конфликт `scope` и параметризации:**
   Обычно `scope="module"` означает, что фикстура создается ровно 1 раз для всего файла. **Но параметризация ломает это правило**. 
   Pytest кэширует фикстуры вместе с их параметрами. Таким образом, экземпляр фикстуры для `"admin"` и экземпляр для `"guest"` — это две разные сущности в кэше Pytest. Поэтому сетап выполнится 2 раза, несмотря на скоуп `module`.

3. **Оптимизация порядка выполнения (Reordering):**
   Кандидат должен упомянуть, что Pytest достаточно умен, чтобы не пересоздавать фикстуры туда-сюда. Если бы Pytest выполнял тесты линейно (read_admin -> read_guest -> write_admin -> write_guest), ему пришлось бы переоткрывать `module`-фикстуру 4 раза. 
   Вместо этого он группирует тесты по параметрам:
   * Создает фикстуру `db_connection(request.param="admin")`.
   * Выполняет `test_read_data[admin]`.
   * Выполняет `test_write_data[admin]`.
   * Выполняет `test_delete_data[admin]` (у этой тестовой функции параметризация только по `["admin"]`).
   * Выполняет teardown для `"admin"`.
   * Создает фикстуру `db_connection(request.param="guest")`.
   * Выполняет `test_read_data[guest]`.
   * Выполняет `test_write_data[guest]`.
   * Выполняет teardown для `"guest"`.

### Что показывает эта задача:
Она мгновенно отсеивает тех, кто знает только теорию («scope=module выполняется один раз для всего файла») от практиков, которые понимают, как Pytest строит граф выполнения, как кэшируются фикстуры в зависимости от `request.param` и как работает механизм `reordering` для оптимизации вызовов (особенно тяжелых, вроде БД или браузеров).
