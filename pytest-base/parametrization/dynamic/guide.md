# Динамическая параметризация и хук `pytest_generate_tests`

> 🟡 **Вопросы (Middle):**
> Перед вами код тестового класса. Для его параметризации используется функция `fetch_users_from_api()`, эмулирующая долгий сетевой запрос или тяжелое вычисление.
> * В какой момент жизненного цикла Pytest выполнится функция `fetch_users_from_api()`?
> * Какие проблемы влечет за собой такой подход?
> * Как переписать этот код, чтобы запрос выполнялся корректно на этапе генерации тестов, а не при импорте модуля?

### Ответ:
Функция выполнится в момент **импорта модуля**, в самом начале фазы сбора тестов.

### Пояснение:

1. **Момент выполнения:**
   В Python декораторы вычисляются в момент импорта модуля интерпретатором. Когда Pytest только начинает сканировать файл `test_dynamic.py`, чтобы найти в нем тесты, интерпретатор Python доходит до строки `@pytest.mark.parametrize("user", fetch_users_from_api())` и **сразу же** выполняет функцию, ожидая получить список.

2. **Проблемы такого подхода:**
   * **Блокировка сбора тестов:** Тяжелый запрос "подвешивает" процесс импорта. Если в этот момент API недоступно или падает с ошибкой, упадёт весь этап сбора тестов. Ни один тест не запустится.
   * **Невозможность конфигурации:** На этапе импорта модуля мы еще не имеем доступа к фикстурам, плагинам или аргументам командной строки Pytest (например, нельзя сделать `if config.getoption("--env") == "prod"`). Данные жестко зашиваются.
   * **Выполнение вхолостую:** Если запустить Pytest с флагом `-k "test_other"` (или запустить только один конкретный тест из файла), функция всё равно отработает, так как файл в любом случае импортируется для анализа.

3. **Как правильно (использование `pytest_generate_tests`):**
   Кандидат должен предложить перенести вызов функции внутрь специального хука `pytest_generate_tests`. Этот хук вызывается фреймворком уже после импорта, непосредственно в момент генерации вариантов для каждой тестовой функции. Пример идеального решения от кандидата:
   ```python
   def pytest_generate_tests(metafunc):
       # Проверяем, нужен ли вообще параметр "user" текущему тесту
       if "user" in metafunc.fixturenames:
           # Теперь мы можем достать конфиг, например: metafunc.config.getoption("--env")
           users = fetch_users_from_api()
           metafunc.parametrize("user", users)
           
   class TestUserProfiles:
       # Декоратор больше не нужен, параметр подцепится из хука
       def test_profile_access(self, user):
           assert isinstance(user, str)
   ```

### Навыки:
Она проверяет, понимает ли автоматизатор разницу между временем импорта Python и временем выполнения/сбора тестов Pytest. Также это базовый вопрос на знание хука `pytest_generate_tests`, который абсолютно необходим при построении динамических Data-Driven фреймворков (чтение данных из БД, API, генерация тестов на основе файлов конфигурации).
